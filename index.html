Chapter 1

Introduction

1.1 Purpose

This Software Requirements Specification (SRS) provides a comprehensive description of the requirements for the Easy Parking System. It establishes a shared understanding between the development team and stakeholders regarding the system behavior, constraints, and quality attributes.

The SRS is the baseline for design, implementation, and testing. The development team uses it to build software that matches user expectations, while stakeholders use it to validate outcomes and request changes when expectations are not met.

1.2 Document Conventions

  • Major sections use hierarchical numbering (for example 1.0, 1.1, 2.3.1).
  • Mandatory requirements are expressed with "shall" and recommendations with "should".
  • Actor names (Administrator, Parking Lot Owner, Car Owner) are capitalized for clarity.
  • Diagram placeholders identify required artifacts to be inserted before final approval.

1.3 Intended Audience and Reading Suggestions

This document is intended for development team members, parking lot owners, car owners, project supervisors, and academic evaluators.

Audience

  • Development team: use to understand requirements and design.
  • Parking lot owners: verify the platform supports operations.
  • Car owners: understand booking, payment, and navigation capabilities.
  • Supervisors and evaluators: review scope and compliance.

Recommended Reading Order

  • Section 1 (Introduction) for all readers.
  • Section 2 (Overall Description) for stakeholders and developers.
  • Sections 3 and 4 (Interfaces and Features) for developers and testers.
  • Section 5 (Non-functional Requirements) for architects and QA engineers.

1.4 Product Scope

The Easy Parking System is a mobile and web platform that automates parking lot operations and helps reduce traffic congestion. It provides real-time availability, online reservation, payment processing, and navigation support.

The solution supports three user roles with role-based access control: Administrator, Parking Lot Owner, and Car Owner.

Chapter 2

Overall Description

2.1 Product Perspective

The Easy Parking System is a self-contained software product designed to replace manual parking management with a digital workflow. It consists of a Flutter mobile client and web interfaces, a Django backend, Firebase authentication, Google Maps integration, and relational database storage.

2.2 Product Functions

User Role Primary Functions Scope
Administrator Full CRUD, audit monitoring, backup and restore, configuration, incident response System-wide
Parking Lot Owner Manage lots, pricing, schedules, alerts, and facility-level reports Own facilities only
Car Owner Search, reserve, pay, cancel, extend booking, receive QR and notifications Personal bookings only

Limited scope applies: parking lot owners manage only their own operational users, and car owners can access only personal booking summaries.

2.3 User Classes and Characteristics

2.3.1 Administrator

  • Full privileges with no restriction across all entities and reports.
  • Manages audit logs, global configuration, backups, and encryption keys.
  • Coordinates external integrations and incident response workflows.

2.3.2 Parking Lot Owner

  • Manages own parking locations, pricing, schedules, and facility settings.
  • Monitors operations through alerts, usage dashboards, and custom reports.
  • Cannot access system-wide administration or other owners' data.

2.3.3 Car Owner

  • Searches and books spaces by location, time, and availability.
  • Pays using cards or digital wallets and receives ticket confirmation.
  • Can cancel or extend reservations and navigate to selected destinations.

2.4 Operating Environment

2.4.1 Hardware Requirements

  • Android or iOS smartphones/tablets for end users.
  • Cloud or on-premise server infrastructure for API and data services.
  • Optional printer hardware for invoices and operational reports.

2.4.2 Software Requirements

  • Flutter client runtime.
  • Django backend with REST endpoints.
  • MySQL or PostgreSQL database engine.
  • Firebase authentication and Google Maps APIs.

2.5 Design and Implementation Constraints

  • Android availability is required; iOS support is planned in future releases.
  • Minimum device memory requirement is 50 MB, with additional memory recommended.
  • The system must remain usable for non-technical users with clear and accessible interfaces.
  • The platform must operate reliably under concurrent load without major degradation.

2.6 Project Documentation

  • Software Requirements Specification (this document).
  • Architecture and database design documentation.
  • Test plan, test cases, and acceptance evidence.
  • Deployment guide and operations handbook.

2.7 Assumptions and Dependencies

  • Users in all roles maintain valid email accounts for notifications.
  • Reliable internet connectivity is available for server and client devices.
  • Google Maps API and Firebase services remain available.
  • Architecture supports future updates and feature expansion post-launch.

Chapter 3

External Interface Requirements

3.1 User Interfaces

Interfaces are role-based with a consistent visual system, responsive layout behavior, and accessibility-aware navigation.

3.1.1 Car Owner Interfaces

  • Search and map discovery screens.
  • Reservation, payment, and booking history workflows.
  • Notification and profile management pages.

3.1.2 Parking Lot Owner Interface

  • Facility dashboards and occupancy controls.
  • Rate configuration and schedule management.
  • Alerts and operational reporting views.

3.1.3 Admin Interface

  • System administration and user governance pages.
  • Audit logs and backup management screens.
  • Platform-level settings and monitoring tools.

3.2 Hardware Interfaces

  • Smartphones and tablets running the Flutter client application.
  • Server infrastructure hosting Django and database services.
  • Laser printer support for bills and formal reports.
  • Broadband networking for real-time communication and email alerts.

3.3 Software Interfaces

  • Django REST API as the central integration layer.
  • Firebase Authentication for user identity and session security.
  • Google Maps services for geocoding and navigation display.
  • Relational database (MySQL or PostgreSQL) for persistence.

3.4 Communications Interfaces

  • HTTPS with TLS 1.2 or higher for all app and backend communication.
  • SMTP over TLS for outbound notifications and reservation events.
  • WebSocket or Firebase channels for real-time availability updates.
  • Minimum 10 Mbps internet recommended for stable map and API performance.

Chapter 5

Non-functional Requirements

5.1 Performance Requirements

  • Availability checks should return within 2 seconds under normal load.
  • Booking confirmation should complete within 5 seconds after payment approval.
  • The platform should support concurrent users without major response degradation.
  • Daily backups must complete within scheduled maintenance windows.

5.2 Safety Requirements

  • Role-based login screens restrict system capabilities by user type.
  • Automated backups and restoration procedures protect data integrity.
  • Sensitive operations (deletion and payment actions) require explicit confirmation.

5.3 Security Requirements

  • All client-server traffic must use HTTPS with TLS 1.2+.
  • Passwords must be hashed using bcrypt or Argon2.
  • Payment processing must satisfy PCI DSS compliance obligations.
  • Admin and owner sessions expire after 30 minutes of inactivity.
  • All APIs enforce validation and protections against SQL injection and XSS.
  • Audit logs must be retained for at least 90 days.

5.4 Software Quality Attributes

Reliability

Predictable behavior with fault-tolerant recovery processes.

Usability

Clear workflows for non-technical users across all roles.

Maintainability

Modular architecture and documented APIs for safe evolution.

Scalability

Capacity to support more users, facilities, and transactions.

5.5 Business Rules

  • Privileges are ordered as Administrator, Parking Lot Owner, then Car Owner.
  • Only Administrators can delete accounts, garages, and access system-wide finance reports.
  • Owners can manage only their own garages and associated operations.
  • Car Owners must complete email verification before the first booking.
  • Refund-eligible cancellation is allowed up to 1 hour before reservation start.
  • Advance payment is required and unpaid bookings expire after 15 minutes.
  • Email notifications are required for confirmation, payment receipt, cancellation, and reminders.
Chapter 6

Other Requirements

  • Training sessions are required for all primary user roles after delivery.
  • Development team support is required for an agreed post-launch period.
  • All clients must maintain active email accounts for system communication.
  • A printed user manual in plain language must be delivered with the system.
  • The system must be monitored post-launch for bugs, performance issues, and enhancements.
